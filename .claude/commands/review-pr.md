# PR 리뷰

백엔드 시니어 개발자 관점에서 PR을 리뷰합니다.

## 인자

$ARGUMENTS - PR 번호 (예: `123`, `#456`)

## 작업

1. **PR 정보 수집**:
   - `gh pr view $ARGUMENTS --json number,title,body,baseRefName,headRefName,files` 로 PR 메타데이터 확인
   - `gh pr diff $ARGUMENTS` 로 전체 diff 확인
   - PR body에서 변경 의도와 맥락을 먼저 파악

2. **변경 파일 분석**:
   - 변경된 파일을 읽고 diff 맥락에서 코드 흐름 파악
   - schema.prisma 변경이 있으면 실제 DB 스키마와 동기화되었는지 확인 (이 프로젝트는 Prisma migration 미사용, 수동 DDL 후 스키마 동기화 방식)
   - DTO 변경이 있으면 기존 API 스펙과의 호환성 확인
   - **import 경로 지적 전 반드시 `tsconfig.json`의 `paths` 설정을 읽어** 실제 alias가 정의된 경로인지 확인 — 정의되지 않은 경로는 alias 대상이 아님
   - **변경된 모듈의 CLAUDE.md를 먼저 읽어** 비즈니스 의도와 설계 결정을 파악 — 인증 가드 누락, 조회 조건 차이 등 의도적인 설계를 오탐으로 지적하지 않도록
   - **환경 분기 코드 지적 전 `.env.example` 또는 배포 설정을 확인해** 실제 사용하는 `NODE_ENV` 값 파악 — 프로젝트별 환경 컨벤션이 다를 수 있음

3. **심각도별 이슈 분류 후 아래 형식으로 출력**

## 리뷰 철학

**핵심 질문**: "이 코드가 지금뿐 아니라 이후에도 (그리고 새로운 작업자가 배정되더라도) 이해/유지 보수가 가능한가?"

- 코드만 보지 말고 **변경 의도**를 이해할 것
- 단순 스타일 지적보다는 **실질적 문제**에 집중
- **왜 이렇게 구현했는지** 맥락을 파악하려 노력할 것

## 리뷰 우선순위

### 🚨 CRITICAL (즉시 수정 필요)

1. **null 참조 가능성**: findUnique 결과를 null 체크 없이 사용, Optional chaining(?.) 미사용
2. **브레이킹 체인지**: Request DTO 필드 제거/타입 변경, Response 구조 변경. 기존 클라이언트(앱, 웹, 파트너사) 영향도 체크
3. **DB 커넥션 풀 소진**: 루프 내부에서 개별 DB 쿼리 실행, Promise.all 없이 순차 처리
4. **Race Condition**: 조회 후 업데이트 사이에 트랜잭션 미사용, 동시 요청 시 데이터 불일치 가능성
5. **컴파일 에러**: duplicate identifier 등, 로컬 서버 실행 여부 확인
6. **런타임 에러**: import되지 않은 module 사용

### ⚠️ HIGH (수정 권장)

7. **인덱스 미적용**: WHERE, JOIN, ORDER BY 컬럼에 인덱스 없음. schema.prisma에 @@index 누락
8. **중복 코드**: 동일/유사 로직 2회 이상 반복, 중복 함수/상수 선언, util 함수 추출 가능성
9. **함수 책임 과다**: 단일 함수 50줄 초과, 3개 이상의 독립적 작업 수행
10. **스키마 동기화 누락**: schema.prisma 변경 시 `prisma validate` / `prisma generate` 미실행, 또는 실제 DB DDL과 schema.prisma 불일치
11. **사이드 이펙트**: 해당 수정사항이 다른 기능에 미치는 영향, 의도하지 않은 동작 변경 가능성
12. **캐싱 가능성**: 자주 변경되지 않을 데이터에 대한 캐싱 미적용
13. **테스트 누락**: 핵심 비즈니스 로직(결제, 포인트 적립/차감 등) 변경 시 테스트 미작성 또는 기존 테스트 미수정

### 💡 LOW (개선 제안)

14. **변수명 불명확**: 단일 문자 변수(i, j 제외), 축약어 과다, 기존 네이밍과의 일관성 부족
15. **문서화 누락**: Swagger 주석, openapi.json, 복잡한 로직 설명, README/CLAUDE.md 업데이트
16. **타입 안정성**: any 타입 사용, as 타입 단언 남용
17. **오타 및 타이포**: 변수명, 함수명, 주석의 철자/문법 오류

## 체크리스트

### 기능 검증
- 의도한 기능이 정확히 구현되었나?
- 예외 케이스가 모두 처리되었나? (빈 배열, null, undefined, 데이터 없음)
- 에러 메시지가 사용자 친화적인가?
- 비즈니스 로직이 요구사항과 일치하는가?

### 성능 및 동시성
- N+1 쿼리 문제가 없나? (include, select 활용)
- 불필요한 순차 처리가 없나? (Promise.all 가능 여부)
- 대용량 데이터 처리 시 페이지네이션이 적용되었나?
- 트랜잭션이 필요한 부분에 적용되었나?
- 비효율적인 동작은 없는가?

### 데이터베이스
- WHERE 절 컬럼에 인덱스가 있나?
- JOIN 대상 외래키에 인덱스가 있나?
- 참조하는 테이블이 적절한가? 외래키 관계가 올바른가?
- schema.prisma 변경이 실제 DB 스키마와 일치하는가?
- DDL 실행 시 락(lock)이 발생하지 않나? (대용량 테이블 ALTER 주의)

### 호환성
- 기존 API 스펙이 유지되나? (필드 제거/타입 변경 확인)
- 연동 클라이언트가 어디인지 파악되었나? 협의가 완료되었나?
- 버전 관리가 필요한 변경인가? (v1, v2)

### 코드 품질
- 중복 로직이 util/helper로 분리되었나?
- 상수가 중복 선언되지 않았나?
- 함수가 단일 책임 원칙을 따르나?
- 변수명이 명확하고 일관적인가?
- 매직 넘버가 상수로 정의되었나?

### 코드 직관성 및 가독성
- 코드 자체만으로 의도가 드러나는가?
- 처음 보는 사람이 맥락 추적 없이 이해 가능한가?
- 과도하게 복잡한 중첩이나 조건문은 없는가?

### 문서화
- Swagger 주석(@ApiOperation, @ApiResponse, @ApiProperty)이 작성되었나?
- openapi.json이 추가/업데이트되었나?
- 복잡한 비즈니스 로직에 주석이 있나?
- README, CLAUDE.md가 최신 상태인가?

### 보안
- 사용자 입력이 검증되나? (DTO validation)
- 민감 정보가 로그에 노출되지 않나?
- 인증/인가가 적절히 적용되었나?

### PR 품질
- PR Body가 `.github/workflows/pull_request_template.md` 형식을 따르는가?
- PR Body가 변경 의도와 범위를 명확히 설명하는가?
- 최초 개발 목적/방향성에서 벗어난 변경사항이 포함되지 않았는가?
- 커밋 메시지가 의미 있게 작성되었는가?
