# 0020. 버전 bump 기준 — patch 기본, 기능 묶음당 minor 1회

- 상태: accepted (ADR 0005 의 단계별 minor 매핑을 supersede)
- 작성일: 2026-06-13

## 맥락

ADR 0005 는 "단계(stage) 완료 = minor" 매핑이었는데 단계 기반 개발은 v0.15 께 끝났고, 이후 PR 단위로 기계적으로 bump 하다 보니 한 작업 묶음에서 0.17.7 → 0.19.0 처럼 minor 가 두 번 뛰는 일이 생겼다 (v0.19.0 은 태그·릴리스 삭제 후 0.18.0 으로 정정). 운영자 피드백: "현재 버전 보고 적절하게 — 0.18.1 → 0.18.2 식으로 가고 기준을 정하자."

## 결정

SemVer `MAJOR.MINOR.PATCH` (메이저.마이너.패치) 에 다음 기준을 적용한다.

| 자리 | 언제 올리나 | 예 |
|------|------------|-----|
| **PATCH** (0.18.0 → 0.18.1) | **기본값.** 버그 수정, 성능 개선, 리팩토링, 프롬프트 튜닝, UI 다듬기, 작은 기능 개선, 의존성·CI | lookback TZ 수정, 수집 캐시, 파일 분할 |
| **MINOR** (0.18.x → 0.19.0) | 사용자가 **체감하는 새 기능 묶음** — 새 화면/탭/설정, 새 수집 소스, 새 워크플로. 릴리스 노트의 주인공이 Features 일 때. **한 작업 묶음에 feat 가 여러 개여도 1회만** | 프롬프트 커스터마이징 탭 + 요약 방향 전환 묶음 = 0.18.0 |
| **MAJOR** (0.x → 1.0.0) | 첫 안정판 선언(서명 배포 등 마일스톤). 1.0.0 이후엔 호환 깨짐(설정/데이터 형식, env, CLI 인터페이스) | — |

운영 규칙:

- docs/CI 만 변경: bump 없음
- 헷갈리면 patch — minor 는 "릴리스 공지하고 싶은가?" 로 판단
- 릴리스(태그 푸시)는 배포하고 싶은 시점에만. 모든 patch 가 릴리스일 필요 없음. 태그 == root `package.json` version (release.yml 이 검증)
- 모듈 버전(`@cairn/core`, `@cairn/desktop`): 같은 기준, 변경된 패키지만 (ADR 0018 의 표시 정책 유지)

## 대안

- **PR 마다 무조건 bump (기존)** — 추적은 쉽지만 버전이 의미 없이 빨리 소모되고 minor 인플레이션 발생
- **release-please 자동화** — Conventional Commits 로 자동 산출 가능하나 feat 마다 minor 를 올려 정확히 이번에 문제가 된 동작을 자동화하게 됨. v1.0.0 이후 재검토

## 결과

- 버전 숫자가 "체감 기능 릴리스" 단위와 일치 — 0.18.1, 0.18.2 … 로 잔잔하게 가다가 기능 묶음에서 0.19.0
- 트레이드오프: "묶음" 판단이 주관적 — 헷갈리면 patch 규칙으로 해소
- ADR 0005 는 superseded (SemVer 사용 자체와 PR template 체크는 유지)
