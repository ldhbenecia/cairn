## 시행착오 / 결정

- **롤업 입력 — 일지 본문 vs 재수집** — daily 페이지의 Summary/Done/InProgress/Notes 블록을 그대로 fetch + 파싱하기로 결정 (재수집 X). 이미 daily 발행 시 sanitize 통과한 한국어 텍스트라 추가 검증 가벼움. 페이지당 1회 `blocks.children.list` 호출 (주간 7회, 월간 ~31회 — Notion rate limit ~3 req/s 안에 충분)
- **`activeRepos` (per-repo breakdown) drop** — 일지 페이지에 repo 단위 카운트가 저장되지 않음 ('Source counts' 는 `gh:N / git:N / notion:N` 합계만). collector 가 신뢰성 있게 채울 수 없는 필드라 contracts 에서 제거. totals 만 유지
- **Notion 클라이언트 분할** — `notion-api.client.ts` 가 worklog + rollup 양쪽 메소드를 다 들고 있으면 비대해짐. → `NotionRollupApiClient` 별도 클래스로 분리, `NotionApiClient.getClient` public 노출해서 client cache 재사용. API I/O 타입은 `notion-api.types.ts` 로 양쪽이 공유
- **ISO week vs 일요일 시작** — ISO 8601 (월요일 시작) 채택. `2026-W17` 표기도 ISO 표준. 한국에서 자연스러움 + plan 의 `2026-W17 주간 정리` 표기와 일관
- **Default `--date` for mode** — plist 의 `ProgramArguments` 가 정적이라 shell `$(date -v-7d)` 못 씀. 앱 쪽에서 mode 별 기본값 (weekly=today-7d, monthly=today-5d) 적용. 두 값 모두 직전 ISO week / 직전 calendar month 안에 들어가는 보장 날짜
- **launchd 시간 — 출근 전 07:00** — 사용자 본인 대부분 일하는 노트북. weekly 는 일요일 daily 마무리(19:00+23:00) 후 충분 마진. monthly 는 매월 1일 daily 마무리 후 마진 1일 → 2일 07:00. 둘 다 11:00 sleep 보강 슬롯 동반
- **활동 0 일 때 skip (rollup)** — `metrics.dailyCount === 0` 이면 summarizer / publisher 호출 안 함. 휴가 기간 / 첫 주 등에서 빈 롤업 발행 안 함 (token / Notion noise 절약). daily skip 정책과 일관
- **롤업 DB 부모 페이지 — worklog 와 같이** — 별도 `rollup.pageId` 강제 안 하고, daily worklog DB 와 같은 부모 페이지 아래에 자동 생성. 사용자 셋업 부담 ↓ (Notion 페이지 하나만 잡으면 됨, daily DB + rollup DB 두 인라인 DB 가 같은 페이지 안에 나란히 들어감). `rollup.pageId` 를 별도로 명시하면 그걸 우선 (다른 페이지 / 워크스페이스 쓰고 싶을 때 override)
- **알림 한국어 — '주간 정리' / '월간 정리'** — 사용자 본인이 보는 거라 한국어 자연스러움. title 'cairn 주간 정리' / 'cairn 월간 정리' / 'cairn 설정 필요' / 'cairn 주간 정리 실패'
